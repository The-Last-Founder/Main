## What Build Is

Build contains:

* A community.

## Principles

### 1. Human over role

Build follows a human-first version of do-ocracy:

> **The project adapts to the human, not the human to the project.**

Inspired partly by [Martin Buber’s I-Thou](https://en.wikipedia.org/wiki/I_and_Thou), contributors are not “resources,” “velocity,” “capacity,” or interchangeable task-fillers.

We try to meet each person as a whole human first, and as a contributor second.

Practical meaning:

* Match quests to energy, mood, life circumstances, and current capacity.
* Make it easy to pause or hand off work without shame.
* Avoid guilt-based recruiting.
* Watch for overwork, addiction loops, and grinding.
* Optimize for sustainable aliveness, not maximum extraction.

Motto:

> **Community over code. Human over community.**

### 2. Guided do-ocracy

People who do the work should have real autonomy.

But do-ocracy does not mean chaos.

It means:

* If you take responsibility for a scoped quest, you get meaningful agency inside that scope.
* If the work affects other people, you coordinate.
* If the decision is reversible, move quickly.
* If the decision is hard to reverse, slow down.
* If safety, conduct, money, legal, or long-term governance is involved, escalate.

### 3. Small core, broad participation

The core team should not become a bottleneck.

Its job is to:

* Keep the space healthy.
* Make contribution paths clear.
* Protect culture.
* Resolve stuck decisions.
* Maintain continuity.
* Help parties form and ship.

The core does not need to own every quest.

### 4. Public by default

Important work should happen in public GitHub surfaces where possible:

* Issues
* Pull requests
* Discussions
* Decision notes
* Markdown docs

Private chat is fine for coordination, but decisions should be summarized publicly when they affect the project.

### 5. Non-code work counts

Build must treat non-code contributions as first-class.

Examples:

* Docs
* Triage
* Design
* Product thinking
* Community care
* Onboarding
* Research
* Testing
* Launch support
* Moderation
* Feedback synthesis
* Sweeping and cleanup

## Operating Model

Build should use **parties** as the basic unit of project work.

A party is a small group gathered around a quest.

A party may include:

* A quest lead
* Prototyper
* Builder
* Sweeper
* Grower
* Maintainer
* Users / testers
* AI agents

A party can be temporary. It does not need to become a permanent team.

A **guild** should be reserved for something more durable, for example:

* Design Guild
* Builders Guild
* Maintainers Guild
* Community Guild
* AI Agents Guild
* Quest Board Guild

Do not create guilds too early. Start with parties. Create guilds only when there is repeated work, shared standards, and recurring ownership.

1. Ron bootstraps as Acting Steward.

3. Maintainers emerge around concrete surfaces.

5. Governance becomes more distributed only when there are enough trusted people to support it.

## Role Architecture

Build uses five AI-native contribution archetypes:

1. **Prototyper**
2. **Builder**
3. **Sweeper**
4. **Grower**
5. **Maintainer**

These are not job titles. They are contribution modes.

A designer can be a Prototyper.
An engineer can be a Sweeper.
A community person can be a Grower.
A PM can be a Builder.
Many people span two roles.

| Archetype  | Core contribution                                                  | Risk                                |
| ---------- | ------------------------------------------------------------------ | ----------------------------------- |
| Prototyper | Creates new ideas, spikes, demos, experiments                      | Too many unfinished ideas           |
| Builder    | Turns ideas into production-grade product, infra, docs, or systems | Ships mess too fast                 |
| Sweeper    | Cleans UI, simplifies systems, removes clutter, improves quality   | Can over-polish or block            |
| Grower     | Improves adoption, onboarding, feedback loops, PMF, community      | Can chase vanity metrics            |
| Maintainer | Keeps mature systems secure, reliable, clear, and alive            | Can become bottleneck or gatekeeper |

### Stage mix

| Stage               | Needed mix                    |
| ------------------- | ----------------------------- |
| New idea            | Prototyper + Builder          |
| Early build         | Builder + Sweeper             |
| Pre-launch          | Sweeper + Grower              |
| Growing             | Grower + Maintainer           |
| Mature              | Maintainer + Sweeper + Grower |
| Community formation | Grower + Maintainer + Steward |

### Core roles

| Role            | Responsibility                                               | Current holder |
| --------------- | ------------------------------------------------------------ | -------------- |
| Acting Steward  | Holds vision, resolves stuck decisions, protects continuity  | Ron            |
| Deputy Steward  | Covers gaps, supports continuity, reduces founder bottleneck | TBD            |
| Quest Lead      | Leads a specific project / quest / party                     | TBD per quest  |
| Maintainer      | Owns a repo, surface, domain, or repeated process            | TBD            |
| Community Guide | Supports onboarding, tone, rituals, and moderation           | TBD            |
| Release Steward | Helps coordinate launches and public releases                | TBD            |
| Contributor     | Participates in scoped work                                  | Open           |

### Role earning

Roles should be earned by contribution and trust, not status.

| Role           | How it is earned                                                |
| -------------- | --------------------------------------------------------------- |
| Contributor    | Shows up respectfully and helps.                                |
| Party member   | Joins a scoped quest.                                           |
| Quest Lead     | Takes responsibility for moving a quest forward.                |
| Maintainer     | Shows repeated judgment, care, review quality, and reliability. |
| Deputy Steward | Demonstrates continuity, judgment, and trust across domains.    |

Roles are not permanent trophies.

If someone becomes inactive:

* Thank them.
* Preserve credit.
* Reduce active ownership if needed.
* Make re-entry easy.

## Decision Rules

Use different decision modes for different decisions.

| Decision type                     | Default mode                           |
| --------------------------------- | -------------------------------------- |
| Small reversible task             | Do it, open PR, get normal review      |
| Work inside owned scope           | Owner / maintainer decides             |
| Cross-scope work                  | Discuss publicly before acting         |
| Governance change                 | Decision issue / RFC                   |
| Safety or conduct issue           | Moderator / safety path                |
| Urgent unowned decision           | Acting Steward decides, then documents |
| Money / legal / irreversible move | Slow down, explicit approval           |

Default rules:

1. **Two-way doors move fast.**
2. **One-way doors slow down.**
3. **Consensus-seeking first, steward tie-break second.**
4. **Record major decisions publicly.**
5. **No meaningful self-merge.**
6. **Human impact counts as part of the decision.**

## Human Sustainability

Build should actively protect people from overwork and project addiction.

This is not a side value. It is operational infrastructure.

Rules:

* Pause without shame.
* Handoff is healthy.
* Tiny quests count.
* Low-capacity work is valid.
* Async is the default.
* No guilt-based recruiting.
* No hero dependency.
* Do not repeatedly tax the same high-energy people.
* If someone becomes indispensable, redesign the system.
* If contribution becomes compulsive, slow down.

Possible future software guardrails:

| Guardrail               | Purpose                                                    |
| ----------------------- | ---------------------------------------------------------- |
| Low-capacity mode       | People can mark themselves available only for tiny quests. |
| Handoff prompt          | Stale issues ask whether to continue, resize, or hand off. |
| Over-contribution nudge | Gentle warning when one person is carrying too much.       |
| Bus-factor check        | Surfaces work that depends on one person.                  |
| Async summary bot       | Reduces pressure to follow every thread.                   |

These should be opt-in and humane, not surveillance.
